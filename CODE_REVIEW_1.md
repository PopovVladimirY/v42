## 4. Schema vs API Gaps

### 4.1 Comments: 7 parent types, 2 implemented

The `comments` table schema (and DB CHECK constraint) supports 7 parent types:
project, epic, backlog_item, task, test, release, stage.

SQL queries only exist for `backlog_item` and `task`. The API routes only expose
comments on backlog items and tasks. Epic/release/stage/test comments are inaccessible
through the API even though the DB schema allows them.

### 4.2 releases and stages tables -- schema only

Created by migration 000002 (or part of the initial schema set). The tables exist
in the database. Zero SQL queries, zero handlers, zero routes.

They are prerequisites for Phase 5 (Timeline). Currently they are DB fixtures with
no application surface.

### 4.3 clarity_level -- field without analytics

`backlog_items.clarity_level` (ENUM: unknown/foggy/explored/clear) exists and is
settable via the backlog CRUD API. The "Clarity Map" and risk-score analytics
planned in Phase 6 have not been implemented. The field is populated but not
analyzed.

### 4.4 Missing migration 000020

Migration sequence skips 000020. Applied migrations go: ...000019, 000021, 000022,
000023, 000024. No file named `000020_*.sql` exists in `migrations/`. This is
cosmetically broken but functionally fine (golang-migrate tracks by filename).

## 5. DESIGN.md vs Reality Divergences

### 5.1 internal/domain/ file list is fictional

DESIGN.md's architecture section lists `internal/domain/` as containing:
`auth.go`, `backlog.go`, `capacity.go`, `stats.go`, `testrun.go`, `sprint.go`,
`team.go`, `skill.go`, `epic.go`, `project.go`.

**Reality:** Only `internal/domain/auth.go` exists. All business logic lives
in the store layer (`internal/db/`) and handlers (`internal/api/`). The domain
layer was never built out beyond auth token types.

### 5.2 Phase 4.5 status table at bottom of DESIGN.md is stale

The "Статус реализации" section near the end of DESIGN.md shows every Phase 4.5
component as "Не начато" (not started). All of them are fully implemented.

### 5.3 Migrations 000021-000024 not in DESIGN.md

Three migrations added features not described in the original design:
- 000021: user_ui_settings
- 000022: retrospective (retro_items, retro_votes, sprints.retro_closed)
- 000023: sprint_capacity (sprint_capacity table)

These are real features with full backend + frontend implementations, but they
were built outside the DESIGN.md phasing plan.

## 6. Code Quality Observations (Positive)

### 6.1 Backend consistency

- Every handler follows `respond()` / `respondErr()` helper pattern -- no raw
  `w.WriteHeader()` calls scattered around.
- Role middleware (`MustRole`, `MustAdmin`) consistently applied in router.go.
- Store interface pattern: handlers depend on `api.Store` interface, not concrete
  types -- correct for testability.
- `notImplemented()` handler helper exists in router.go but is **not used on any
  route**. This is a good sign: no ghost routes.

### 6.2 Auth implementation

- JWT access tokens: 15-minute expiry (short and correct).
- Refresh tokens: httpOnly cookie, 7-day expiry, stored in DB for revocation.
- Passwords: bcrypt (cost factor sufficient).
- Token rotation on refresh: old token invalidated on use.
- Rate limiting middleware present in router.go.

### 6.3 Frontend patterns

- Consistent modal routing: detail pages (BacklogItemDetailPage, TaskDetailPage,
  TestDetailPage) rendered as route-based overlays with ESC to close. Pattern
  applied uniformly.
- TanStack Query used consistently for all server state. No manual fetch in
  components.
- CSS custom properties only -- no hardcoded colors, no Tailwind utility classes
  for color (only for layout/spacing). Themes work by swapping the var() values.
- TypeScript strict mode on. All 29 pages type-check clean.

### 6.4 Error handling

- Backend: `respondErr()` always wraps with status + message. No bare 500s found.
- Frontend: React Query error states propagated, but no global error boundary.
  Individual pages handle errors inline.

## 7. Minor Issues

### 7.1 useIdleTimeout: warning is console.warn, not UI toast

`useIdleTimeout.ts` schedules a `console.warn()` 1 minute before auto-logout.
The user has no visible warning in the UI. A toast or modal confirmation is needed
to match the intent of the feature.

**Priority:** Low (feature works, UX is weak).

### 7.2 Two idle hooks -- clear separation

`useIdleDetect` (ambient art trigger) and `useIdleTimeout` (auto-logout) share
the same event listener pattern. They are separate hooks with different purposes
and both are used in `AppShell.tsx`. This is correct, not a bug. Just note that
any future "reset idle" logic must be mirrored in both.

### 7.3 DnD reorder: backend done, frontend not connected

Backend has `POST /api/v1/projects/{id}/backlog/reorder` endpoint. The backlog
page (`BacklogPage.tsx`) does not implement drag-and-drop reordering. Mentioned
in TODO.md.

### 7.4 pushRecentProject: never called

`pushRecentProject()` from `useLastProject.ts` is exported but grep shows no
calls to it in any page or shell component. The sidebar shows recent projects,
but they are never pushed -- the list stays empty until manually saved.

**Action:** Call `pushRecentProject(user.id, projectId, projectName)` in
`ProjectShell.tsx` on mount.
