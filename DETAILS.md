## PostgreSQL -- база данных

Это реляционная БД. Данные хранятся в таблицах со строками и столбцами, как в очень умном Excel.
Связи между таблицами -- через foreign keys (ссылки по ID).

**Почему Postgres, а не MySQL / SQLite / MongoDB:**

- **SQLite** -- файловая БД, нет нормального параллельного доступа. Для PM-инструмента
  с несколькими пользователями -- не вариант.
- **MySQL** -- хорошая, но Postgres умнее в сложных запросах, типах данных (JSONB, массивы,
  enums), и честнее соблюдает стандарты SQL.
- **MongoDB** -- документная БД (хранит JSON-документы). Для нашей модели с кучей связей
  между сущностями реляционная модель лучше. Не надо изобретать JOIN вручную.

Postgres -- промышленный стандарт уже 30 лет. Не устаревает, не предаёт.

## golang-migrate -- миграции схемы

**Проблема:** структура БД меняется со временем. Добавили поле, переименовали таблицу,
добавили индекс. Как это делать безопасно и воспроизводимо?

**Решение:** миграции -- это пронумерованные SQL-файлы. Каждый файл описывает одно изменение.

```
migrations/
  000001_init_schema.up.sql     -- применить изменение
  000001_init_schema.down.sql   -- откатить изменение
  000002_add_skills.up.sql
  000002_add_skills.down.sql
```

`golang-migrate` запускает их по порядку и запоминает что уже применено -- хранит версию
прямо в БД. Новый разработчик клонирует репо, запускает `make migrate-up` -- и получает
ровно ту же схему что у всех остальных.

Никаких "а у меня локально другая база".

**Правило:** никогда не редактируем уже применённую миграцию. Нужно исправить -- пишем
новую миграцию. Это не баг, это дисциплина.

## sqlc -- типизированный SQL без ORM

Самый необычный инструмент в стеке. Объясняем подробно.

**Проблема с ORM (Hibernate, EF Core, Prisma):**
Ты пишешь на языке ORM, ORM генерирует SQL. Ты не видишь какой SQL выполняется.
Появляется N+1 проблема (1000 запросов вместо одного), магические join'ы которые работают
не так как ты думал. DBA смотрит в query log и плачет.

**Проблема с сырым SQL в Go без sqlc:**
Пишешь строку `"SELECT * FROM users WHERE id = $1"`, результат приходит как `interface{}`,
и сам руками парсишь строки в структуры. Опечатался в названии колонки -- узнаешь только
в рантайме, когда всё уже упало в продакшене.

**sqlc решает это элегантно:**

Шаг 1 -- пишешь обычный SQL файл с аннотацией:
```sql
-- name: GetUserByEmail :one
SELECT id, email, display_name, role, is_active
FROM users
WHERE email = $1 AND is_active = true;
```

Шаг 2 -- запускаешь `sqlc generate` (или `make sqlc`).

Шаг 3 -- получаешь готовый типизированный Go-код:
```go
func (q *Queries) GetUserByEmail(ctx context.Context, email string) (User, error) {
    // sqlc написал это за тебя. Типы правильные. Колонки правильные.
}
```

Опечатался в SQL -- sqlc не скомпилируется и скажет где именно.
SQL читаемый, оптимизируемый, лежит в git как обычный файл.
Никакой магии -- только SQL и Go.

**Аннотации sqlc:**
- `:one`  -- ожидаем одну строку (или ошибку если нет)
- `:many` -- ожидаем список строк
- `:exec` -- не ожидаем результата (INSERT, UPDATE, DELETE)

## chi -- HTTP роутер

Go имеет встроенный `net/http` -- он уже умеет принимать запросы. Но для реального API нужны:
- URL-параметры: `/projects/{id}/epics` -- достать `id` из пути
- Группировка маршрутов с общим middleware
- Цепочки middleware: auth -> roles -> logging -> handler

`chi` добавляет именно это, и ничего лишнего.

**Сравнение роутеров:**

| Критерий            | net/http | chi           | Gin      | Echo     |
|---------------------|----------|---------------|----------|----------|
| Размер              | встроен  | маленький     | средний  | средний  |
| URL params          | вручную  | встроено      | встроено | встроено |
| Middleware chains   | вручную  | чисто         | чисто    | чисто    |
| Магия               | нет      | нет           | немного  | немного  |
| Совместимость       | стандарт | 100% стандарт | своя     | своя     |

Gin и Echo используют свои типы для handlers -- привязка к фреймворку.
`chi` работает со стандартными Go handler'ами: `http.HandlerFunc`.
Если chi завтра умрёт -- меняем роутер за день, весь остальной код не трогаем.

**Пример группировки маршрутов:**
```go
r := chi.NewRouter()
r.Use(middleware.Logger)       // логируем все запросы

r.Post("/api/v1/auth/login", authHandler.Login)

r.Group(func(r chi.Router) {
    r.Use(authMiddleware.Authenticate) // только авторизованные
    r.Get("/api/v1/projects", projectHandler.List)
    r.Post("/api/v1/projects", projectHandler.Create)

    r.Group(func(r chi.Router) {
        r.Use(roleMiddleware.RequireAdmin) // только admin
        r.Post("/api/v1/users", userHandler.Create)
    })
})
```

## JWT -- авторизация без состояния

**Как работает:**

1. Пользователь логинится: `POST /auth/login` с email + password
2. Сервер проверяет пароль через bcrypt, создаёт **токен** -- подписанная строка:
   `header.payload.signature`
3. Payload содержит: `user_id`, `role`, срок жизни (`exp`)
4. Клиент сохраняет токен и отправляет в каждом запросе:
   `Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...`
5. Сервер проверяет подпись -- если валидна, знает кто это и какая роль. **Без обращения к БД.**

**Два токена -- зачем:**

- **Access token** -- живёт 15 минут, используется для каждого запроса
- **Refresh token** -- живёт 7 дней, хранится в БД, используется только для продления сессии

Если access token утёк -- через 15 минут он умрёт сам.
Refresh token можно отозвать (logout -- удаляем из БД).
Это баланс между удобством и безопасностью.

**bcrypt -- для паролей:**
Пароли никогда не хранятся в открытом виде. bcrypt -- односторонняя хэш-функция
с встроенной солью. Даже если утечёт вся БД -- пароли не расшифруются.

## SSE -- Server-Sent Events (real-time)

**Проблема:** Иван переместил карточку на доске. Петру нужно увидеть это без перезагрузки.

**Варианты решения:**

| Подход    | Как работает                              | Минусы                        |
|-----------|-------------------------------------------|-------------------------------|
| Polling   | Клиент спрашивает каждые 5 сек "что нового?" | Расточительно, задержка 0-5с |
| WebSocket | Двусторонний постоянный канал             | Сложнее в реализации и ops    |
| SSE       | Сервер держит HTTP open, шлёт события     | Только сервер -> клиент       |

SSE -- это обычный HTTP GET, который не закрывается. Сервер пишет в него текстовые события:
```
data: {"type":"item_status_changed","id":"abc","status":"in_progress"}

data: {"type":"item_moved","id":"def","stage_id":"xyz"}
```

Браузер получает их через стандартный `EventSource` API:
```typescript
const events = new EventSource(`/api/v1/projects/${projectId}/events`, {
    headers: { Authorization: `Bearer ${token}` }
});
events.onmessage = (e) => {
    const event = JSON.parse(e.data);
    // обновляем UI
};
```

Для PM-инструмента нам нужно только "сервер -> клиент" (карточки двигают люди, сервер
сообщает об этом остальным). SSE -- идеальный fit. Работает через nginx без дополнительной
настройки. WebSocket был бы overkill.

## Docker Compose -- запускаем всё вместе

Инструмент для запуска нескольких сервисов как одного целого.

```
docker compose up
```

Запускает:
1. **postgres** -- БД, изолированная, со своим томом (данные живут между перезапусками)
2. **api** -- скомпилированный Go бинарник, знает адрес postgres из env vars

Новый человек в команде: `git clone` -> `cp .env.example .env` -> `docker compose up` ->
всё работает. Без установки Go, без установки PostgreSQL, без танцев с PATH.

**Dev vs Prod:**
- `docker-compose.yml` -- базовая конфигурация
- `docker-compose.dev.yml` -- для разработки: volume mounts для hot-reload, adminer для
  просмотра БД в браузере, порты открыты для дебага

```
make dev    -- docker compose -f docker-compose.yml -f docker-compose.dev.yml up
make prod   -- docker compose up -d
```

## Полная картина -- как всё работает вместе

```
Браузер
  |
  |-- React (UI, состояние, роутинг)
  |-- dnd-kit (drag-and-drop карточек)
  |-- fetch() -> REST API запросы
  |-- EventSource -> SSE поток событий
  |
  V  HTTP / HTTPS
Go API (единый бинарник)
  |
  |-- chi router (куда идёт запрос)
  |-- Logger middleware (записываем все запросы)
  |-- Auth middleware (проверяем JWT токен)
  |-- Role middleware (есть ли права?)
  |-- Handler (бизнес-логика)
  |-- sqlc (выполняем типизированный SQL)
  |
  V  TCP (5432)
PostgreSQL 16
  |
  |-- Данные на диске (Docker volume)
  |-- Индексы для быстрых запросов
  |-- Транзакции для атомарных операций
```

Всё это живёт в Docker Compose. На сервере -- один `docker compose up`. Локально -- то же самое.
