## Доменная модель (что строим)

Прежде чем стек -- надо понять форму данных. Иначе выбор стека -- пальцем в небо.

### Иерархия рабочих элементов

```
Project
  |-- Epic
  |     |-- BacklogItem (Story / Bug / Feature)
  |           |-- Task
  |           |     |-- Test
  |           |-- Test
  |-- Test (acceptance, project-level)

TestReport (cross-cutting: по проекту, эпику, беклог-айтему)
```

Важный момент: тесты висят на нескольких уровнях. Это не баг, это фича -- acceptance tests живут
на уровне BacklogItem, integration/system tests -- на уровне Epic или Project. Нужна единая
модель теста с полиморфной привязкой (parent_type + parent_id) или отдельные junction-таблицы.

### Таймлайн

```
Project
  |-- Release (дата начала, дата конца, цель)
        |-- Milestone / Stage (этап)
              |-- BacklogItems (привязка к этапу)
```

Release -- это не Sprint. Sprint -- это итерация команды (2 недели). Release -- это публичное
обязательство (квартал, полгода). Этап (Stage/Milestone) -- промежуточная веха внутри релиза.
Планирование нагрузки живёт на пересечении: кто, сколько часов, в каком этапе.

### Роли и команды

```
User
  |-- GlobalRole: Admin | Maintainer | Developer | Tester | Observer
  |-- Team membership:
        |-- TeamRole: Lead | Member
        |-- Specialization: Frontend | Backend | QA | DevOps | Design | ...
```

Специализация -- это не роль в системе, это компетенция. Один человек может быть Backend + DevOps.
Нагрузка считается по специализации: "нам нужно 40 часов backend в этом этапе, у нас есть 60 --
окей". Это отвечает на требование "план должен быть реалистичным".

### Нагрузка (Load Planning)

```
TeamMember
  |-- capacity_per_sprint: hours (например, 32 часа из 40 с учётом митингов)
  |-- assignments:
        |-- Task (estimate_hours, actual_hours)
        |-- Stage allocation (зарезервировано часов в этапе)
```

Плановая нагрузка = sum(task estimates) по члену команды в периоде.
Факт = sum(actual_hours) по закрытым задачам.
Отклонение -- вот тебе и статистика.

## Эпики -- отдельная история

В V1 эпик был неудобным существом: есть, но непонятно где живёт, как связан со спринтом,
как отображается в прогрессе. Делаем по-человечески:

**Эпик = контейнер с собственным lifecycle.**

- У эпика есть статус: Draft / Active / Done / Cancelled
- У эпика есть owner (не только команда)
- Прогресс эпика = автоматически: % закрытых беклог-айтемов (по count и по estimate)
- Эпик живёт на timeline наравне с Release -- это его "горизонт"
- Epic Board: отдельный вид, где видны все эпики проекта, их статус, прогресс, команда
- Эпик может span несколько релизов (большой эпик -- это нормально)
- Беклог-айтем без эпика -- это тоже нормально (технический долг, hotfix и т.д.)

## Варианты стека

### Вариант 1 -- "Народный фронт" (Node.js + React)

**Backend:** Node.js + Fastify + TypeScript
**Frontend:** React 19 + TypeScript + Vite
**База:** PostgreSQL 16
**ORM:** Drizzle ORM (легче Prisma, SQL-first, отличная TypeScript поддержка)
**Auth:** JWT (access + refresh tokens), bcrypt для паролей
**Real-time:** Server-Sent Events (SSE) для live-обновлений (проще WebSocket, достаточно для нас)
**Drag-and-drop:** dnd-kit (лучший на сегодня для React, гибкий, a11y из коробки)
**Timeline view:** планировать на basis of custom SVG или взять vis-timeline
**Deployment:** Docker Compose (postgres + backend + frontend через nginx)

```
v42/
  apps/
    api/          -- Fastify + Drizzle
    web/          -- React + Vite
  packages/
    db/           -- schema, migrations (Drizzle)
    shared/       -- общие типы (TypeScript)
  docker-compose.yml
  .env.example
```

**Плюсы:**
- Огромная экосистема -- для любой хотелки найдётся библиотека
- TypeScript сквозь весь стек -- типы shared между backend и frontend
- Fastify быстрее Express в 2-3 раза из коробки
- Drizzle -- SQL без магии, миграции прозрачные
- dnd-kit -- лучшее drag-and-drop решение для React прямо сейчас
- Любой JS-разработчик зайдёт в проект без раскачки

**Минусы:**
- Node.js под нагрузкой требует аккуратности (но для PM-инструмента нагрузка смешная)
- Monorepo требует настройки (pnpm workspaces -- решается)
- JS-экосистема меняется быстро (зато Fastify + Drizzle -- стабильные зрелые проекты)

---

### Вариант 2 -- "Серьёзные дяди" (ASP.NET Core + React)

**Backend:** ASP.NET Core 9 (C#) + Web API
**Frontend:** React 19 + TypeScript + Vite (тот же, что и в варианте 1)
**База:** PostgreSQL 16
**ORM:** Entity Framework Core + Npgsql
**Auth:** ASP.NET Identity + JWT Bearer
**Real-time:** SignalR (WebSocket с fallback, от Microsoft -- работает надёжно)
**Drag-and-drop:** dnd-kit (без изменений)
**Deployment:** Docker Compose (postgres + dotnet api + nginx + frontend)

**Плюсы:**
- C# -- строгая типизация, отличная производительность (быстрее Node.js в большинстве сценариев)
- SignalR -- battle-tested real-time, работает из коробки
- EF Core -- мощные migrations, LINQ-запросы без SQL
- .NET ecosystem стабильнее JS -- зависимости не ломаются каждый год
- Swagger/OpenAPI -- документация API генерируется автоматически

**Минусы:**
- C# + React -- два разных языка, два разных мира, переключение контекста
- EF Core миграции бывают "с сюрпризом" при сложных схемах
- Чуть тяжелее в Docker (образ больше, cold start медленнее)
- Меньше C# разработчиков в open-source web-сообществе

---

### Вариант 3 -- "Элегантный минимализм" (SvelteKit + Node.js)

**Full-stack:** SvelteKit (server-side rendering + client hydration)
**API:** встроенные SvelteKit server routes + отдельный Fastify для тяжёлых операций
**База:** PostgreSQL 16
**ORM:** Drizzle ORM
**Auth:** Lucia v3 (отличная для SvelteKit)
**Real-time:** SSE (встроено в SvelteKit без боли)
**Drag-and-drop:** svelte-dnd-action (легче dnd-kit, native для Svelte)
**Deployment:** Docker Compose (node + postgres)

**Плюсы:**
- SvelteKit -- это server + client в одном, меньше кода, меньше boilerplate
- Svelte быстрее React по умолчанию (нет virtual DOM)
- Меньше файлов, меньше конфигурации
- SSR из коробки -- SEO и первый рендер быстрый (для PM-инструмента не критично, но приятно)
- Один язык (TypeScript) на весь стек

**Минусы:**
- Svelte меньше React по размеру сообщества -- меньше готовых компонентов
- Timeline/Gantt библиотеки для Svelte слабее, чем для React
- Drag-and-drop для сложных board-сценариев менее зрелый чем dnd-kit
- Если проект вырастет и придут контрибьюторы -- React они знают, Svelte -- не факт

## Рассуждения по интересным идеям

### 1. Компетенции, уровни и рост -- живой человек в системе

Это самая недооценённая область в PM-инструментах. Jira знает о тебе только твой логин.
VersionOne знала чуть больше. V.42 должен знать, кто ты профессионально -- и уважать это.

**Модель компетенции:**

```
TeamMember
  |-- skills: [
  |     { skill: "TypeScript", level: Expert,    interest: High  },
  |     { skill: "React",      level: Expert,    interest: High  },
  |     { skill: "PostgreSQL", level: Competent, interest: Medium },
  |     { skill: "Python",     level: Beginner,  interest: High  }
  |   ]
  |-- capacity_per_week: 32h  (рабочие часы минус митинги)
```

Уровни: Beginner / Competent / Proficient / Expert (4 ступени -- достаточно, не надо 10).
Интерес: Low / Medium / High -- это не каприз, это инвестиция в будущее.

**Эффективный коэффициент при планировании задачи:**

Задача требует Python. Назначаем Beginner с High interest + есть Expert рядом (mentoring).

```
effective_capacity = base_hours * level_factor * (1 + mentoring_bonus)
```

- Expert:    level_factor = 1.0
- Proficient: 0.8
- Competent:  0.6
- Beginner:   0.4  (но с mentoring от Expert => +0.2 бонус)

Это не абсолют -- это ориентир для планирования. Не надо делать из этого NASA.
Главное: система знает, что Beginner в Python не то же самое, что Expert, и не позволяет
тебе тихо "вкатить" 8-часовую задачу в питоне новичку как будто это нормально.

**Рост и история:**

Каждая закрытая задача -- это факт. После N задач в компетенции система может предложить:
"Иван закрыл 12 Python задач за последние 2 месяца, возможно стоит пересмотреть уровень?"
Не автоматически повышать -- это должно быть решением тимлида, но система напоминает.

**Почему это важно на уровне системы:**

- Планировщик видит реальную картину нагрузки, а не фиктивную
- Тимлид видит рост команды в цифрах, а не на ощущениях
- Mentoring становится видимым -- его можно планировать и ценить
- Команда развивается управляемо, а не стихийно

Jira -- антипример: там человек это строка в таблице. V.42 -- человек это актив с динамикой.

### 2. Многомерность: Эпики, Релизы, Этапы -- мухи отдельно, котлеты отдельно

Ключевое прозрение: **Epic и Release -- это не иерархия, это два независимых измерения.**

Epic -- это "ЧТО мы строим" (продуктовое измерение).
Release -- это "КОГДА мы это выпускаем" (временное измерение).

BacklogItem живёт на пересечении обоих, но не обязан быть в обоих одновременно:

```
BacklogItem
  |-- epic_id:     nullable  (может быть без эпика -- технический долг, hotfix)
  |-- stage_id:    nullable  (может быть без привязки к этапу -- в backlog-буфере)
  |-- release_id:  nullable  (может быть без релиза -- ещё не запланировано)
```

Все три поля независимы. Это не parent-child. Это теги-измерения.

**Визуально это выглядит так:**

- Вид "по эпикам": показываем прогресс каждого эпика независимо от релизов
- Вид "по релизам": показываем что войдёт в релиз независимо от эпиков
- Вид "по этапам": показываем нагрузку этапа -- что из каких эпиков туда попало
- Вид "backlog": всё, что не привязано ни к чему -- кандидаты на планирование

Такая модель позволяет честно отвечать на вопросы:
- "Сколько беклог-айтемов из Эпика X попадёт в Релиз 2.0?" -- CROSS-фильтр
- "Что из Этапа 3 относится к Эпику Y?" -- тот же CROSS-фильтр
- "Покажи всё незапланированное" -- фильтр по null

**Эпик как горизонт, не как контейнер:**

Epic не "содержит" беклог-айтемы в смысле ownership. Epic -- это цель-горизонт.
Беклог-айтем "относится к" эпику -- как статья относится к теме.
Это принципиально меняет восприятие и снимает искусственные ограничения V1.

### 3. Команда как живой организм -- нормализация статистики

Проблема с raw-статистикой очевидна: в январе в команде 3 человека, в марте -- 7.
Сравнивать "часов в январе vs часов в марте" -- бессмысленно без нормализации.

**Что нормализуем и как:**

```
Velocity (нормализованная) = story_points_done / available_capacity_in_period
```

Где available_capacity = sum(capacity_per_week * weeks) по всем активным членам команды
в данный период. Это даёт реальную производительность, а не абсолютные цифры.

**По компетенциям:**

```
Competency throughput = tasks_done_in_skill / available_hours_in_skill_for_period
```

Это отвечает на вопрос: "насколько эффективно мы используем наши Python-компетенции в этом
квартале?" Не "сколько Python задач сделали" (это абсолют), а "сколько из возможного".

**Как не переборщить:**

Правило простое: показываем только то, что влияет на решение. Три уровня статистики:

1. **Оперативная** (для тимлида еженедельно): кто сколько взял, кто перегружен, кто простаивает
2. **Тактическая** (для планирования релиза): velocity команды, throughput по компетенциям
3. **Стратегическая** (для ретроспективы): рост компетенций, тренды, что не учли в эстимейтах

v1 делаем только первые два уровня. Третий -- когда данных накопится достаточно.
Не надо строить аналитическую платформу в первой версии. Сначала данные, потом анализ.

### 4. Трудозатраты: Node.js vs .NET Core -- честная оценка

Условие: есть команда, которая умеет и Node.js, и .NET. Спрашивается -- сколько времени
от "пустая папка" до "рабочий продукт v1" по каждому варианту?

**Что входит в v1 (для оценки):**
- Авторизация + роли
- CRUD для всех сущностей (Project, Epic, BacklogItem, Task, Test, User, Team)
- Sprint board с drag-and-drop
- Timeline (Release + Stage) с базовой визуализацией
- Capacity planning (простая версия)
- Базовая статистика
- Docker Compose деплой

**Вариант 1 -- Node.js + Fastify + React + Drizzle**

| Этап                          | Оценка    |
|-------------------------------|-----------|
| Настройка monorepo (pnpm ws)  | 1-2 дня   |
| DB schema + Drizzle migrations| 2-3 дня   |
| Auth (JWT + роли)             | 2-3 дня   |
| API endpoints (все сущности)  | 8-10 дней |
| React: базовые CRUD views     | 5-7 дней  |
| Sprint board + dnd-kit        | 4-6 дней  |
| Timeline view                 | 5-8 дней  |
| Capacity planning UI          | 3-5 дней  |
| Статистика                    | 2-3 дня   |
| Docker + CI/CD                | 1-2 дня   |
| **Итого**                     | **33-49 дней** |

Реалистично для одного full-stack разработчика: **7-10 недель.**
Для двух (backend + frontend): **4-6 недель.**

**Вариант 2 -- ASP.NET Core + React**

| Этап                          | Оценка    |
|-------------------------------|-----------|
| Настройка solution (.NET)     | 1 день    |
| DB schema + EF Core migrations| 2-4 дня   |
| Auth (Identity + JWT)         | 3-4 дня   |
| API controllers (все сущности)| 8-12 дней |
| React: базовые CRUD views     | 5-7 дней  |
| Sprint board + dnd-kit        | 4-6 дней  |
| Timeline view                 | 5-8 дней  |
| Capacity planning UI          | 3-5 дней  |
| Статистика                    | 2-3 дня   |
| Docker + CI/CD                | 2-3 дня   |
| **Итого**                     | **35-53 дня** |

Реалистично для одного разработчика (C# backend + React frontend): **8-11 недель.**
Для двух (backend C# + frontend React): **4-6 недель.**

**Вывод по трудозатратам:**

Разница небольшая -- 5-15% в пользу Node.js. Это не тот разрыв, который определяет выбор.
Определяет другое:

- **Node.js** быстрее стартует (меньше церемоний), но у него "семь пятниц" в экосистеме --
  за год-два зависимости могут устареть, и обновление будет стоить времени.
- **.NET Core** медленнее на старте (больше scaffolding, C# vs JS переключение с React),
  но стабильнее на длинной дистанции. Microsoft не бросает .NET -- это факт.
  Плюс: C# + EF Core даёт меньше "WTF" при сложных запросах, чем Drizzle + TypeScript.

**Если думать на 3+ года:** .NET Core выглядит надёжнее как фундамент.
**Если думать на "запустить и посмотреть":** Node.js быстрее до первого работающего demo.

Нет плохого варианта -- есть разные ставки.

## Вариант 4 -- "Методичный и чистый" (Go + React + PostgreSQL)

**Решение принято:** Node.js наелись, C# наелись. Go -- новый, но правильный.
Бонус: хорошая причина познакомиться поближе с языком, который не предаёт.

### Стек

**Backend:**  Go 1.22+ + chi (router) + sqlc (type-safe SQL codegen) + golang-migrate
**Frontend:** React 19 + TypeScript + Vite + dnd-kit
**База:**     PostgreSQL 16
**Auth:**     JWT (golang-jwt/jwt/v5) -- никакого Identity framework, всё прозрачно
**Real-time:** SSE (Server-Sent Events) -- встроено в Go без доп. зависимостей
**Деплой:**   Docker Compose: один бинарник Go + postgres + nginx

### Почему Go закрывает оба болевых точки

**Против "семи пятниц" Node.js:**
- Стандартная библиотека покрывает 80% задач -- HTTP, JSON, crypto, testing -- всё встроено
- Сторонних пакетов минимум: chi, sqlc, golang-migrate, golang-jwt -- и всё
- `go.sum` криптографически верифицирует каждую зависимость -- supply chain атаки мимо
- Go 1.x backward compatible с Go 1.0 (2012). Язык не меняется. Нет breaking changes.

**Против церемоний .NET:**
- Компилируется в один статический бинарник: скопировал -- работает
- Docker-образ: 15-20MB (scratch + бинарник) против 300MB Node и 200MB .NET
- Никакого runtime на сервере. Никаких dll. Никакого GAC.
- Нет EF Core магии с миграциями -- только честный SQL через sqlc

### sqlc -- это отдельная философия

Пишешь SQL. Получаешь типизированный Go-код. Без ORM. Без магии.

```sql
-- query.sql
-- name: GetBacklogItemsByEpic :many
SELECT id, title, status, estimate_hours, epic_id, stage_id, release_id
FROM backlog_items
WHERE epic_id = $1 AND project_id = $2
ORDER BY priority;
```

Запускаешь `sqlc generate` -- получаешь готовый Go-метод с правильными типами.
Никаких N+1 запросов которые ты не видишь. Никаких "а что он там под капотом делает".
SQL читаемый, оптимизируемый, версионируемый. DBA скажет спасибо.

### Структура проекта

```

  internal/
```

## API -- не как в V1, а по-человечески

VersionOne API был известен тем, что туда страшно было заходить без защитного снаряжения.
Собственный query language, нестандартные URL, ответы которые надо было парсить с молитвой.
V.42 делаем иначе.

### Принципы API

**REST, но думающий.**
Не "REST для галочки", а ресурсно-ориентированный дизайн. Каждый URL -- это ресурс.
Методы -- HTTP-глаголы по смыслу. Никаких `POST /api/doSomethingWeird`.

```
GET    /api/v1/projects                     -- список проектов
POST   /api/v1/projects                     -- создать проект
GET    /api/v1/projects/{id}                -- один проект
PATCH  /api/v1/projects/{id}                -- обновить (только что изменилось)
DELETE /api/v1/projects/{id}                -- удалить

GET    /api/v1/projects/{id}/epics          -- эпики проекта
POST   /api/v1/projects/{id}/epics          -- создать эпик в проекте

GET    /api/v1/backlog-items?epic={id}      -- фильтр по эпику
GET    /api/v1/backlog-items?release={id}   -- фильтр по релизу
GET    /api/v1/backlog-items?stage={id}     -- фильтр по этапу
GET    /api/v1/backlog-items?unplanned=true -- ничем не привязанные
```

**OpenAPI / Swagger из коробки.**
Документация генерируется из кода -- не пишется вручную и не устаревает.
Любой клиент может сгенерировать SDK автоматически. Postman-коллекция в один клик.

**Версионирование сразу.**
`/api/v1/...` с первого дня. Когда придётся сломать совместимость -- будет `/api/v2/...`.
Старая версия живёт параллельно. Никаких "мы всё сломали, обновляйтесь срочно".

**Предсказуемые ответы.**
Всегда одна структура:
```json
{
  "data": { ... },       // или [] для списков
  "meta": {              // для списков: pagination
    "total": 42,
    "page": 1,
    "per_page": 20
  },
  "error": null          // или { "code": "NOT_FOUND", "message": "..." }
}
```
Никаких сюрпризов. Клиент всегда знает что ожидать.

**PATCH, не PUT.**
Обновление только изменившихся полей. Не надо слать весь объект чтобы изменить один статус.
Это критично для drag-and-drop: переместил карточку -- отправил `{ "stage_id": 42 }`. Всё.

**Фильтрация, сортировка, пагинация -- стандартные.**
```
GET /api/v1/backlog-items?status=open&sort=priority&order=asc&page=2&per_page=50
```
Никакого собственного query language как в V1. Query params -- просто и понятно.

**SSE endpoint для real-time:**
```
GET /api/v1/projects/{id}/events   -- stream событий проекта
```
Клиент подписывается, получает обновления когда кто-то двигает карточки или меняет статус.
Без WebSocket сложности, без polling. Работает через обычный HTTP.

### Что это даёт

- Внешние интеграции пишутся за часы, не за дни
- Мобильный клиент когда-нибудь -- тот же API, ничего нового
- Автоматизация через скрипты -- curl + jq, и всё понятно
- Сторонние инструменты (Zapier, n8n, etc.) -- подключаются стандартно
- CI/CD интеграции -- статус задачи из pipeline, время как у людей

## Kafka как событийный хаб

**Статус**: надо подумать. Не включено в план реализации.
**Актуально для**: Phase 7 (SSE) и далее.
**Инфраструктура**: `kv8-kafka` уже крутится в Docker -- не гипотетика.

### Проблема которую решаем

У V.42 три разных потребности в "истории":
1. **Хронология проекта** -- что изменилось, кто сделал, когда. Activity feed.
2. **Real-time уведомления** -- SSE, @mentions, назначения.
3. **Интеграции** -- CI/CD автозапуск тестов, Slack, webhooks.

PostgreSQL справляется с #1 (через `activity_log` таблицу).
С #2 -- при одном инстансе (через горутину-broadcaster).
С #3 -- нет.

Kafka решает все три, но добавляет операционную сложность.

### Предлагаемая граница: SQL vs Kafka

**В PostgreSQL остаётся** (текущее состояние, ACID, сложные запросы):
- Все entity-таблицы: users, projects, epics, backlog_items, sprints...
- `comments` -- редактируемые, пагинируемые, FK-целостность
- `sprint_test_results` -- текущий статус для отчётов и JOIN
- `notifications` -- is_read, счётчик непрочитанных
- `activity_log` -- materialized из Kafka (consumer пишет сюда)
- `outbox` -- transactional outbox буфер (ключевой паттерн)

**В Kafka уходит** (история, fan-out, интеграции):

| Topic | Тип событий | Зачем |
|-------|-------------|-------|
| `v42.events.items` | status_changed, assigned, reordered | SSE fan-out; velocity analytics |
| `v42.events.sprints` | started, completed, item_added | CI/CD интеграция |
| `v42.events.tests` | result_recorded, cascade_skipped | Audit trail |
| `v42.events.comments` | created, edited, deleted | @mention уведомления |
| `v42.notifications` | любые user-level уведомления | Push к конкретному юзеру |
| `v42.audit` | копия всего, long retention | Compliance, replay, debug |

### Ключевой паттерн: Transactional Outbox

Проблема: "записали в БД, Kafka упала -- событие потеряно".
Решение: в одной транзакции с основной записью пишем в `outbox`, горутина-publisher
асинхронно льёт из outbox в Kafka. At-least-once delivery, idempotency key в payload.

```sql
CREATE TABLE outbox (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    topic        TEXT NOT NULL,
    key          TEXT NOT NULL,        -- partition key (project_id)
    payload      JSONB NOT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    published_at TIMESTAMPTZ           -- NULL = не отправлено
);
CREATE INDEX idx_outbox_unpublished ON outbox(created_at) WHERE published_at IS NULL;
```

### SSE с Kafka vs без

| | Без Kafka (Phase 7 план) | С Kafka |
|-|--------------------------|---------|
| Broadcaster | горутина внутри процесса | Kafka consumer в каждом инстансе |
| Масштабирование | один инстанс | горизонтально, без sticky sessions |
| Сложность | низкая | средняя |
| Нужно сейчас | нет | нет |

### Что НЕ трогаем

`comments` остаются в SQL -- они редактируемые, Kafka immutable, не совместимо
как хранилище. В Kafka летит только event `comment.created` с body_preview для уведомлений.

### Альтернативы Kafka

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| PostgreSQL LISTEN/NOTIFY | Уже есть, без доп. зависимости | Только один инстанс, нет персистентности |
| Redis Pub/Sub | Легче Kafka | Нет персистентности, нет replay |
| NATS JetStream | Легче Kafka, персистентность | Меньше экосистема |
| Kafka | Персистентность, replay, экосистема | Операционная сложность |

### Вывод

Не сейчас. `outbox` таблицу заложить в следующую миграцию -- ничего не ломает, просто ждёт.
Kafka или NATS JetStream -- решить в Phase 7 когда делаем SSE. Тогда же понятно,
нужен ли горизонтальный масштаб или хватит одного инстанса.

- [ ] Kafka vs NATS JetStream: выбрать к Phase 7
- [ ] outbox таблица: заложить в следующую миграцию?

## Квадранты недопонимания -- пятое измерение планирования

Классическая матрица знания (Рамсфелд, Джохари) даёт четыре квадранта.
У неё есть проблема: она предполагает, что всё уже разобрано.
В реальности большинство беклог-айтемов живут в пятом квадранте -- лимбе непереработанного.

```
                       ЗНАЮ              НЕ ЗНАЮ
               +------------------+------------------+
  ОСОЗНАЮ      |  1. Явное знание  | 3. Известный риск |
               |  (clear)         | (scoped)          |
               +------------------+------------------+
  НЕ ОСОЗНАЮ   |  2. Скрытое знание| 4. Черный лебедь  |
               |  (tacit)         | (foggy)           |
               +------------------+------------------+

  [5. Лимб] -- unclassified: ещё не разобрано. Всё что прилетело и ждёт тriage.
```

### Описание квадрантов

**Квадрант 1 -- Явное знание (clear).**
Знаем что делать, знаем как проверить, знаем кто делает.
Acceptance criteria заполнены и команда с ними согласна.
Берётся в спринт без вопросов.

**Квадрант 2 -- Скрытое знание (tacit).**
Компетенция в команде есть, но не задокументирована. "Васян знает как это работает,
спроси Васяна". Bus factor = 1. До старта спринта нужен knowledge transfer или паринг.
Хуже чем кажется: если Васяна нет -- задача зависает.

**Квадрант 3 -- Известный риск (scoped).**
Open questions идентифицированы. Знаем, чего не знаем -- это уже половина победы.
Нужен spike, исследование или уточнение с заказчиком. Риск управляемый.
В спринт можно, но с буфером времени на "разведку".

**Квадрант 4 -- Черный лебедь (foggy).**
Много неизвестного, и мы даже не знаем, каких вопросов мы не задаём.
Брать в спринт без отдельного spike -- самострел.
Сначала spike (таймбоксированное исследование), потом планирование.

**Квадрант 5 -- Лимб (unclassified).**
Пришло, записали, не разобрались. Это не дефект -- это нормальное состояние нового беклог-айтема.
Но: оставаться в лимбе долго -- опасно. Каждый нетронутый лимбовый айтем накапливает
"clarity debt" -- долг по ясности, который рано или поздно придётся платить.

Правило grooming: цель каждого груминга -- вытащить айтемы из лимба в квадранты 1-3.
Квадрант 4 (foggy) -- сигнал для отдельного исследовательского задания.

### Связь с моделью Дрейфуса

Дрейфус описывает вертикаль компетентности: Beginner -> Novice -> Competent -> Proficient -> Expert.
Квадранты описывают горизонталь ясности: foggy -> tacit/scoped -> clear.

Их пересечение даёт неожиданные выводы:

- **Expert в квадранте 4** -- самый опасный тип риска. Старший разраб уверен,
  что "всё понятно", а за углом архитектурная пропасть которую он не видит
  потому что слишком давно работает в этой области. Confidence bias.
- **Beginner в квадранте 3** -- лучшее что может быть. Понимает собственные пробелы,
  задаёт правильные вопросы, не делает уверенных заявлений.
- **Competent в квадранте 2** -- типичная ситуация: человек умеет, но объяснить не может.
  Знания пока на уровне рефлексов, не на уровне модели.

Вывод: уровень экспертизы не гарантирует ясность. Проверять надо отдельно.
Дрейфус -- кто ты. Квадранты -- что ты знаешь об этом конкретном айтеме.

### Применение к планированию

Каждый элемент планирования (проект, эпик, беклог-айтем, задача, тест) получает поле
`clarity_level`. Это не оценка качества работы и не статус выполнения. Это оценка
степени понимания на момент планирования.

```
clarity_level ENUM:
  'unclassified' -- квадрант 5: пришло, ещё не разобрали
  'foggy'        -- квадрант 4: много неизвестного, риск высокий
  'tacit'        -- квадрант 2: знания есть, но только у одного человека
  'scoped'       -- квадрант 3: open questions идентифицированы
  'clear'        -- квадрант 1: всё понятно, можно брать
```

Практические правила:
- В спринт: только `clear` и `scoped` (с буфером)
- На груминг: `unclassified` и `foggy` -- обязательные кандидаты
- Spike-задача: создаётся для `foggy` айтемов перед их планированием
- Алерт: `tacit` айтем без носителя знания в текущем спринте -- стоп-фактор

### Связь с ATDD-моделью

Acceptance criteria (ac_setup, ac_steps, ac_expected) -- это не просто документация.
Это основной инструмент снижения тумана.

```
AC не заполнены         => clarity = 'unclassified' или 'foggy' автоматически
AC заполнены            => clarity может быть 'scoped' (open questions остались)
AC заполнены + команда согласна => clarity = 'clear' (готово к спринту)
```

Формула: ясность = AC + команда которая с ними согласна + носитель знания доступен.

Это не правило движка -- это принцип для команды. Система может подсветить несоответствие
(ясность = clear, но AC пустые), но не обязана блокировать. Доверяем людям, не роботизируем.

### Применение к скил-сету команды

Квадранты работают не только для айтемов, но и для знаний команды:

```
Skill dimension map (командный уровень):
  clear:        команда знает что умеет, умеет применять, может передать знания
  tacit:        умеет, но знания у одного (bus factor >= 1 -- это риск)
  scoped:       знает границы незнания (нужно обучение, есть план)
  foggy:        думает что умеет, но с критическими пробелами (самый опасный)
  unclassified: новая технология или домен, пока никто не оценивал
```

Это мета-уровень knowledge map команды. Не "кто что умеет по Дрейфусу",
а "насколько команда осознаёт свои знания и незнание как систему".

Пример: команда "умеет Kubernetes" (так написано в профилях), но на деле
один человек понимает как оно работает изнутри, остальные копируют конфиги.
Без квадрантного анализа система видит "4 человека знают Kubernetes" -- оптимистичная ложь.
С квадрантным анализом: skill_clarity = 'tacit', bus_factor_alert = true.

### Оценка риска спринта через квадранты

```
Sprint risk score = (foggy_count * 4 + unclassified_count * 3 +
                     tacit_count * 2 + scoped_count * 1 + clear_count * 0)
                    / total_items

Низкий риск:   score < 0.5  (>80% айтемов clear/scoped)
Средний риск:  0.5 - 1.5    (есть foggy, но немного)
Высокий риск:  > 1.5        (foggy или unclassified на критическом пути)
```

Это не светофор для блокировки -- это сигнал для разговора на planning meeting.
"У нас risk score 2.1 в этом спринте. Давайте посмотрим на foggy айтемы прежде чем брать."

### Практический lifecycle айтема по квадрантам

```
Product manager пишет идею => unclassified
         |
         v
  Grooming session:
    - поняли что не понятно => scoped (open questions записаны)
    - обнаружили много тумана => foggy => создаём spike
    - всё ясно сразу => clear
    - знает только один человек => tacit => планируем pairing
         |
         v
  Spike выполнен => переоцениваем clarity
  Knowledge transfer => tacit => clear или scoped
         |
         v
  Sprint planning => только clear и scoped берём в спринт
         |
         v
  Done => clarity зафиксирована в истории (что не знали в начале, поняли по ходу)
```

История изменений clarity_level по айтему -- это ценные данные для ретроспективы.
"Этот айтем провёл 3 недели в foggy до того как пошёл в спринт -- почему?"

### Почему это важно для V.42

Большинство PM-инструментов фиксируют только то, что сделано.
V.42 должен фиксировать и то, что не было понято вовремя.

Это основа надёжных решений: не обязательно лучших, но надёжных.
Пространство для манёвра сохраняется именно там, где туман убирается заранее,
а не когда он неожиданно материализуется в середине спринта.

Квадранты -- не бюрократия. Это честный разговор с собой перед тем как взять обязательства.
