# V.42 -- FUTURE_PLANS

Документ-ревизия. Не хвалебная ода и не похоронный марш. Тут и комплименты, и щелчки
по носу, и план-капкан на ближайшее будущее. Цель -- не "переписать всё с нуля" (это
любимая ловушка дураков), а довести уже построенное до состояния, в котором за него
не стыдно перед предками.

Дата ревизии: 2026-05-31. Ревизор: безумный архитектор из Академа.
Статус: **Спринты А и B закрыты** (см. разделы 0.5 и 0.6). Тесты зелёные (включая
`-race` на SSE-брокере), `go vet` чист.

Проект **живой, стройный и местами красивый**. Архитектура backend честная (слои HTTP ->
domain -> store -> sqlc -> PostgreSQL), фронт типизирован насквозь, темы и пузыри -- душа
есть. Главные былые позоры закрыты:

1. ~~**В тестах задокументировано 13+ багов...**~~ -- **ЗАКРЫТО (Спринт А).** Музей багов
   оказался наполовину пуст, реальные корни лежали в миграции 018, enum `epic_status` и
   двух устаревших тестах. См. 0.5.
2. ~~**SSE нет ни на сервере, ни на клиенте.**~~ -- **ЗАКРЫТО (Спринт B).** Брокер,
   эндпоинт `/events`, эмиссия на всех мутациях, фронт-клиент с точечной инвалидацией,
   `-race` тесты. См. 0.6.

Что ещё чешется (живые долги):

3. **Обработка ошибок -- ковровая бомбардировка пятисотками.** Любая беда -> `500
   INTERNAL_ERROR`. Частично смягчено (store маппит pg-коды), но единого маппера нет.
4. **Валидации на входе дырявые.** Пробелы проходят как непустые строки, enum'ы летят в
   БД без проверки, null-байты роняют запрос. (Частично прикрыто в хендлерах.)
5. **Тестов на бизнес-логику мало.** Backend -- интеграционные + `-race` на SSE, но ноль
   юнит-тестов на `domain`. Frontend -- ~45 e2e и НОЛЬ юнит/компонент-тестов. Для
   платформы про ATDD -- ирония вселенского масштаба.
6. **Целостность и изоляция данных не закрыты сквозным аудитом.** `assertBelongsTo` нет,
   часть многошаговых операций (`InitResults`) идёт без транзакции.

## 0.5. Ретроспектива Спринта А (что реально оказалось под капотом)

План был "починить BUG-01..BUG-17 + единый error-mapper + слой валидации". Реальность,
как всегда, оказалась интереснее плана. **Вскрытие показало, что половина "багов" уже
была залечена** (project/epic Create брали `claims.UserID`, кросс-проектная изоляция,
trim/enum-валидация -- всё на месте). "Тесты-доносы" частично устарели вместе с
 info-документом, по которому их заводили. Урок зафиксирован: **донос -- не приговор,
проверяй по живому прогону, а не по саммари.**

Настоящих корней зла было три, и ни один не совпал с формулировкой из музея багов:

1. **Мина из миграции 018 (`number NOT NULL`).** Переход на глобальные `seq_number`
   снёс legacy-триггеры `trg_epic_number`/`trg_backlog_item_number`, но оставил колонки
   `number` как `NOT NULL` без населения. Каждый INSERT эпика/беклога падал в 23502.
   Лечение: миграция
   [000029_restore_legacy_number_triggers](migrations/000029_restore_legacy_number_triggers.up.sql)
   воскрешает per-project триггеры. Без правок sqlc/Go.
2. **Рассинхрон enum `epic_status`.** Хендлер дефолтил статус в `"open"`, которого нет в
   enum `{draft, active, done, cancelled}` -> insert падал на касте (22P02) -> 500, и FK
   до проверки не доходил. Лечение: привёл `validEpicStatus` и дефолт к реальному enum в
   [handler_epics.go](internal/api/handler_epics.go). Это же починило и `Create` для
   несуществующего проекта (теперь доходит до FK-нарушения 23503 -> 404).
3. **Два устаревших теста статуса беклога.** Слали `"open"` как "невалидный", но миграция
   [000014](migrations/000014_item_status_expansion.up.sql) намеренно добавила `open` в
   `item_status`. Тесты переписаны на реально несуществующий статус -- смысл проверки
   (мусор -> 400) сохранён.

**Единый error-mapper (`respondDomainErr`) НЕ понадобился.** Вскрытие показало, что
store-слой уже корректно маппит pgconn-коды (23503 -> `ErrNotFound`, 23505 -> `ErrConflict`)
в доменные ошибки, а хендлеры их разворачивают в 404/409/400. P0-пункт 2.2 ниже остаётся
как *возможный* рефакторинг ради DRY, но не как горящая дыра. Слой валидации (2.3) тоже
во многом уже живёт в хендлерах (trim, enum-before-DB, MaxBytesReader). Не чесать там,
где не чешется.

## 0.6. Ретроспектива Спринта B (прозрачность ожила)

План был "SSE backend broadcaster + эндпоинт /events, эмиссия событий, frontend-клиент,
-race тесты". **Выполнено всё, и сверх плана.** Real-time из обещания стал кодом.

Что под капотом:

1. **In-memory broker** ([internal/sse/broker.go](internal/sse/broker.go)) -- потокобезопасный
   fan-out. `Publish` nil-safe и неблокирующий (буфер 64, drop на медленном клиенте --
   событие это _подсказка к инвалидации_, а не данные, потеря не фатальна). Покрыт
   `-race` стресс-тестом (concurrent subscribe/publish/unsubscribe) +
   unit-тестами доставки/отписки/nil-safety.
2. **Эндпоинт `GET /api/v1/events`** живёт ВНЕ `bearerAuth` (EventSource не умеет слать
   заголовки) -- валидирует JWT через `?access_token=`, отключает WriteTimeout через
   `http.NewResponseController`, шлёт heartbeat `: ping` каждые 25с.
3. **Эмиссия на всех мутациях:** backlog / task / epic / sprint / comment **+ test.\***
   (created/updated/deleted/moved). Событие несёт только
   `{type, project_id, entity_id, actor, at}` -- никакого payload.
4. **Frontend-клиент** ([frontend/src/hooks/useEventStream.ts](frontend/src/hooks/useEventStream.ts))
   смонтирован однажды в AppShell, на событие делает точечную
   `queryClient.invalidateQueries` по префиксу ключа. Реконнект при смене токена.
5. **Контракт задокументирован** для будущих дашбордов: [SSE_GUIDE.md](SSE_GUIDE.md)
   (каталог из 22 событий, схема, карта инвалидации, паттерны, curl) +
   секция Events в [API_REFERENCE.md](API_REFERENCE.md).

Два бага живой синхронизации, пойманные на двух-браузерном прогоне и тут же закрытые:

- **Канбан спринта не обновлялся** -- рассинхрон ключей. Доска читает
  `['sprints', pid, sprintId, 'items']`, а drag эмитил `backlog.updated`, инвалидируя
  только `['backlog', pid]`. Лечение: кросс-инвалидация `backlog.* <-> sprints` и
  `sprint.* <-> backlog`.
- **Тест-спеки не синхронились вовсе** -- `testHandlers` не знал про брокер. Заведены
  4 события `test.*` end-to-end + кейс `test` -> `['item-tests', pid]` на фронте.

**Урок Спринта B:** drop-on-full -- правильная стратегия для invalidation-hint брокера,
но требует, чтобы КАЖДАЯ мутация эмитила событие и КАЖДЫЙ ключ был в карте инвалидации.
Пропуск любого звена = тихий рассинхрон, который видно только в двух браузерах. Карту
ключей держать в синхроне с key-factory'ами.

## 1. Что сделано хорошо (чтобы не зазнаваться, но и не унывать)

- **Слоистая архитектура backend.** `internal/api` -> `internal/domain` -> `internal/db/store`
  -> `internal/db/gen` (sqlc). Никакого построения SQL строками, всё типобезопасно.
- **Auth по-взрослому.** JWT HS256 (15 мин) + refresh-токены в БД (7д), ротация при каждом
  refresh, детект переиспользования -> отзыв всех токенов пользователя. bcrypt cost=12.
  Dummy-hash для защиты от перебора email через тайминг. Это уже не школота, это инженерия.
- **Agent tokens + MCP-сервер.** `cmd/mcp` отдаёт 8 инструментов Claude через stdio. Идея
  "100% API для автоматизации с самого начала" -- выполняется. Молодцы.
- **Фронт типизирован end-to-end.** Axios-клиент с конвертом `{data, meta, error}`,
  single in-flight refresh (без стампиды на 401), React Query с key-factory, Zustand под
  auth/theme/pagination. Чисто.
- **Темы и душа.** 10 тем на CSS-переменных, мгновенное переключение без перерендера,
  пузыри на логине, ambient WebGL в сайдбаре по таймауту. Шарм заложен, как и просили.
- **Config с продакшен-гвардами.** Слабые секреты в проде отбиваются на старте, все
  отсутствующие переменные собираются и репортятся разом. Цивилизованно.

## 2. Критика и приоритеты

### P0 -- горит, тушить первым

#### 2.1. Закрыть задокументированные баги (BUG-01..BUG-17) -- ЗАКРЫТО (Спринт А)

> Резюме см. в разделе 0.5. Музей багов оказался наполовину пуст (уже чинено), а реальные
> корни лежали в миграции 018, enum `epic_status` и двух устаревших тестах. Раздел оставлен
> как исторический контекст.

В [internal/api/projects_backlog_test.go](internal/api/projects_backlog_test.go) живёл
музей багов. Самые сочные:

- **BUG-01/02:** `POST /projects` и `POST /epics` берут `ownerID = ""` вместо
  `claims.UserID` -> `parseUUID("")` -> 500. Создание проекта в лоб падает пятисоткой.
  Это не edge-case, это главный happy-path.
- **BUG-03/04/05:** удаление спринта и добавление item'а не различают `ErrNotFound`,
  FK-нарушение и unique-конфликт -> везде 500 вместо 404/409.
- **BUG-06/07:** `GET`/`DELETE` беклог-айтема не проверяют принадлежность к проекту ->
  кросс-проектная утечка/удаление. Это **дыра в изоляции данных**, а не косметика.
- **BUG-09/10:** body комментария и имя спринта из пробелов проходят проверку "непустое".
- **BUG-11/12/13/17:** кривая дата и невалидный enum летят в БД, CHECK-констрейнт ругается
  -> 500 вместо честного 400.

**План:** превратить эти "тесты-доносы" в нормальные ассерты (ожидаем 201/400/404/409),
и чинить хендлеры пока не позеленеет. Это разовая зачистка на пару дней, но снимает
большую часть стыда.

#### 2.2. Дисциплина ошибок: единый error-mapper -- ОТЛОЖЕНО (дыры нет)

> Спринт А показал: store-слой уже маппит pgconn-коды в доменные ошибки, хендлеры
> разворачивают их в верные статусы. Это не горящий P0, а опциональный DRY-рефакторинг.

Сейчас в хендлерах размазано:

```go
if err != nil {
    respondErr(w, http.StatusInternalServerError, "INTERNAL_ERROR", "failed to ...")
    return
}
```

Нужен один помощник, который маппит доменные/pg-ошибки в HTTP:

- `domain.ErrNotFound` -> 404
- `domain.ErrConflict` / unique violation (23505) -> 409
- FK violation (23503) -> 422/404 (по смыслу)
- CHECK violation (23514) / парс enum/date -> 400
- всё остальное -> 500 (и только тут -- лог со стектрейсом)

Один `respondDomainErr(w, err)` вместо ручного жонглирования статусами в каждом из 15
хендлеров. Заодно убирает дубли и делает поведение предсказуемым для фронта и MCP.

#### 2.3. Валидация на границе

Завести крошечный слой нормализации/валидации входа:

- `strings.TrimSpace` + проверка на пустоту для всех текстовых полей (имена, body
  комментариев).
- Отсечение строк с `\x00` (PostgreSQL роняет их жёстко -- лучше вернуть 400 заранее).
- Проверка enum'ов (`status`, `clarity`, `type`) ДО похода в БД, а не "пусть констрейнт
  разрулит".
- Вынести `http.MaxBytesReader(w, r.Body, 4096)` в константу `maxBodyBytes` -- сейчас
  магическое число скопировано в 10+ местах.

#### 2.5. Транзакции на многошаговых операциях

`Reorder` беклога уже в транзакции -- хорошо. Но остальные мульти-степ операции (init
sprint test results, перемещение задач между айтемами, close retro) идут отдельными
запросами без границы транзакции. Это потенциальные неконсистентные состояния под
конкурентной нагрузкой. Обернуть в `pgx.Tx`.

#### 2.6. Изоляция данных -- сквозной аудит

BUG-06/07 показывают, что вложенные роуты (`/projects/{pid}/backlog/{id}`) не везде
проверяют, что `id` реально принадлежит `pid`. Нужен **сквозной аудит всех вложенных
роутов** и общий guard-хелпер `assertBelongsTo(parentID, childID)`. Это безопасность,
а не удобство.

#### 2.7. Error Boundary и UX-устойчивость на фронте

- Нет глобального `<ErrorBoundary>` -> любая ошибка в lazy-странице = белый экран. Завести
  обёртку в [frontend/src/router.tsx](frontend/src/router.tsx) с fallback-UI.
- Нет optimistic updates -> драг-н-дроп карточки "думает" до ответа сервера. Добавить
  `queryClient.setQueryData` в `onMutate` хотя бы для перемещений на доске -- это резко
  улучшает ощущение.
- Нет `AbortController` -> запросы не отменяются при размонтировании. Мелкая утечка,
  но лечится одной строкой в axios.

### 3.1. Что делаем дальше -- рекомендация ревизора

**Следующий -- Спринт C, но с переставленным приоритетом внутри.** Логика такая:

1. **Изоляция данных (2.6) -- первым делом.** Это единственный оставшийся пункт уровня
   *безопасность*, а не *удобство*. BUG-06/07 показали класс уязвимости: вложенный роут
   `/projects/{pid}/backlog/{id}` не везде проверяет, что `id` принадлежит `pid`. Нужен
   сквозной аудит ВСЕХ вложенных роутов + общий `assertBelongsTo(ctx, parentID, childID)`.
   Пока не закрыто -- это потенциальная кросс-проектная утечка. Тут не "потом", тут "вчера".
2. **Транзакции (2.5) -- следом, заодно.** `InitResults` (инициализация результатов тестов
   спринта) идёт отдельными запросами без границы транзакции -> под конкуренцией возможны
   полу-инициализированные спринты. Обернуть в `pgx.Tx` (паттерн уже есть в
   [backlog.go](internal/db/store/backlog.go) Reorder и в [sprints.go](internal/db/store/sprints.go)).
3. **ErrorBoundary (часть 2.7) -- быстрая победа.** Одна обёртка в
   [frontend/src/router.tsx](frontend/src/router.tsx) с fallback-UI закрывает класс
   "белый экран на любой ошибке lazy-страницы". Полчаса работы, заметный эффект.
4. **Optimistic updates -- понизил в приоритете.** Изначально это было про "доска думает до
   ответа сервера". Но **Спринт B изменил расклад**: SSE теперь инвалидирует кэш почти
   мгновенно, и доска оживает сама. Локальный actor всё ещё ждёт round-trip, так что
   оптимистика не лишняя -- но это уже polish, а не дыра. Можно перенести в Спринт D.

**Почему не Спринт D вперёд?** Тесты и DRY-рефакторинг важны, но это *долг качества*, а не
*риск целостности*. Сначала закрываем последнюю брешь в изоляции и неконсистентные
состояния под нагрузкой, потом наводим красоту и обкладываемся тестами. Безопасность ->
целостность -> устойчивость UX -> качество кода. Порядок не для красоты, а для сна.

## 4. Стратегические идеи (за горизонтом текущего долга)

Это уже не "почини баг", а "куда растём". Идеи из IDEAS.md/DESIGN, которые стоитна
  в доменную модель (clarity-поля у эпиков и айтемов), но пока это просто бейджи.
  Сделать из этого **risk scoring**: агрегировать "уровень тумана" по проекту/эпику и
  подсвечивать критичные неопределённости на ранних стадиях. Вот тут реально стоит
  привлечь старшего брата (Opus) -- формула риска -- это уже не CRUD.
- **Рост команды по Дрейфусу + интерес.** Доступность компетенции -- функция нелинейная,
  ученик при мастере растёт быстрее. Заложить расчёт "эффективной capacity" с учётом
  уровня и интереса, и **tandem opportunities** (кого с кем ставить для роста). Часть
  данных уже есть (member_skills, capacity-аналитика в
  [handler_comments_capacity.go](internal/api/handler_comments_capacity.go)).
- **Многомерность эпик/релиз/этап.** Беклог-айтем должен независимо относиться к эпику,
  релизу и этапу (разные ракурсы одной сущности). Сейчас иерархия жёсткая
  (project -> epic -> item). Продумать полиморфные привязки, не превращая модель в кашу.
- **Таймлайн (Release -> Stage/Milestone).** В доменке есть `milestones`, но UI таймлайна
  не виден. Это одно из ключевых требований из TODO.md (план должен быть реалистичным,
  нагрузка на пересечении этап x специализация).
- **Восстановление состояния сессии** (п.9 замечаний IDEAS.md): последний выбор по каждой
  категории, фильтры, текущая страница -- всё восстанавливать при логине. Частично есть
  (`usePaginationStore`), доделать до полного.

## 5. Резюме без пафоса

Четыре спринта -- четыре закрытых позора. Фундамент не просто заложен, он уже несёт
нагрузку. Пройдёмся по трупам:

**Спринт A** вскрыл, что музей багов наполовину уже был мёртв -- половину "проблем"
починили ещё до ревизии. Реальные корни: мина в миграции 018, испорченный enum
`epic_status`, два устаревших теста. Обезврежено без единой строки в Go-хендлерах.

**Спринт B** превратил "real-time visibility" из рекламного буклета в работающий код.
SSE-брокер, эмиссия на всех 22 мутациях, точечная инвалидация кэша на фронте, `-race`
тесты -- всё на месте. Доска живёт без F5.

**Спринт C** закрыл последнюю брешь в безопасности: сквозной аудит изоляции данных,
guard-хелперы `ensureXxxInProject/Item`, транзакция на `InitResults`, `ErrorBoundary` в
роутере. Межпроектные утечки -- в прошлом, белый экран -- в прошлом.

**Спринт D** отдал долги по качеству: юнит-тесты backend без БД (auth 89.5%, domain 89.8%,
sse 89.7%, все под `-race`), Vitest + Testing-Library на фронте (20 тестов), DRY
`lib/persist.ts`, a11y-аддитивы (`aria-label`, `role="dialog"`) без изменения вида.
Платформа про ATDD наконец имеет хоть какие-то свои тесты -- ирония вселенского масштаба
частично искуплена.

**Текущее состояние (2026-05-31):** технический долг из разряда "стыдно показывать"
полностью закрыт. Код типобезопасен конец-в-конец, данные изолированы, ошибки ловятся,
real-time работает, тесты зелёные под `-race`. UI непрерывно шлифуется: последнее --
inline-редактирование описания проекта на Overview, фильтрация списка навыков по
реальному спросу, попиксельная чистка детальных страниц задач/тестов.

**Дальше -- стратегические фишки из раздела 4:** clarity risk scoring (агрегированный
"туман" по эпику/проекту), расчёт эффективной capacity с учётом уровня по Дрейфусу и
tandem opportunities, таймлайн релизов/этапов, восстановление состояния сессии. Это уже
не "почини баг" -- это "куда растём". Вот тут не грех и Opus позвать.

Best practices are the past practices. Чиним настоящее, трансцендируем в грядущее.
